## 1. 우리는 AI를 쓰고 있지만, AI와 함께 일하고 있지는 않다

### 8명이 각자 AI를 쓰고 있다

우리 팀은 이미 Claude Code + Git을 활발히 사용하고 있다. 8명의 팀원이 참여하고, 33개 스킬이 만들어졌으며, 17개 이상의 분석 프로젝트가 진행됐다. **도구 도입은 끝났다.**

하지만 솔직히 말하면, 지금 우리는 **"8명이 각자 AI를 쓰는 팀"**이지 **"AI로 함께 일하는 팀"**은 아니다.

### 세 가지 문제

**1. 지식이 단절된다**
- 내가 인터뷰 분석에서 배운 교차검증 방법을 옆 사람은 모른다
- 누군가 발견한 Athena 쿼리 함정을 다른 사람이 똑같이 밟는다
- 프로젝트가 끝나면 노하우가 머릿속에만 남고, 다음 프로젝트는 다시 0에서 시작한다

**2. 품질이 들쭉날쭉하다**
- 같은 인터뷰 분석을 해도 산출물 구조가 팀원마다 다르다
- 데이터 분석의 기본 규칙(`event_type = 'PAYMENT'` 필터, GMV 기준 등)이 공유되지 않아 같은 실수가 반복된다
- "누가 분석했느냐"에 따라 결과의 깊이가 달라진다

**3. 사람이 모든 걸 기억해야 한다**
- `payment_type` 컬럼은 실제로 존재하지 않는다는 것, 할인금액은 음수로 저장된다는 것, 로그 테이블은 `category` 컬럼을 써야 한다는 것... 이런 걸 매번 사람이 기억하고 있어야 한다
- Claude Code는 매 세션마다 "이 프로젝트가 뭔지" 처음부터 추론해야 한다. 어제의 맥락을 오늘의 세션이 모른다

### 이 제안의 핵심

"AI를 각자 쓰는 것"에서 **"팀이 함께 AI로 일하는 것"**으로 전환하자.

구체적으로는 세 가지 변화다:
1. **"각자 AI" → "함께 AI"**: 내가 배운 것이 자동으로 팀에 전달되는 구조
2. **"지식이 사라지는 구조" → "지식이 쌓이는 구조"**: 프로젝트를 할수록 팀이 더 똑똑해지는 구조
3. **"사람이 기억" → "시스템이 기억"**: 규칙, 원칙, 교훈을 사람 대신 시스템(CLAUDE.md, LESSONS.md)이 기억

새로운 도구를 도입하는 것이 아니다. **이미 쓰고 있는 Claude Code가 팀의 지식을 자동으로 읽고 따르게 만드는 것**이다.

## 2. 이미 바뀐 사례, 그리고 우리가 바뀔 수 있는 방향

### 2.1 Before & After — 실제로 일어난 변화

6개월간 13개 이상의 프로젝트를 Claude Code로 진행하면서, "시스템이 기억하는 구조"를 직접 만들고 검증했다. 그 전과 후의 차이다.

#### 인터뷰 분석

| | Before | After |
|---|---|---|
| **프로세스** | 매번 "이런 구조로 분석해줘" 지시 | CLAUDE.md에 7개 섹션 구조가 적혀 있어 "P3 분석해줘"만으로 표준 산출물 생성 |
| **데이터 검증** | 인터뷰 발언을 그대로 인사이트로 채택 | 6-7번 섹션에서 Athena 구매 데이터로 교차검증 자동 수행 |
| **소요 시간** | 2-3시간/건 | 30-40분/건 (~75% 절감) |
| **품질** | 분석자마다 다름 | 누가 해도 같은 구조, 같은 깊이 |

#### 데이터 분석

| | Before | After |
|---|---|---|
| **쿼리 실수** | `event_type` 필터 누락, `payment_type` 컬럼 호출 등 반복 | CLAUDE.md에 규칙이 적혀 있어 Claude Code가 자동 적용 |
| **맥락 전달** | 세션마다 "이 테이블은 이렇게 생겼고..." 설명 반복 | 프로젝트 CLAUDE.md가 맥락을 기억. 설명 불필요 |
| **레슨런** | 실수 후 "다음엔 주의해야지" (머릿속) | LESSONS.md에 자동 기록. 다음 세션에서 Claude Code가 읽고 적용 |

#### 프로젝트 시작

| | Before | After |
|---|---|---|
| **셋업** | 폴더 구조, 분석 방법, 쿼리 규칙을 매번 새로 정리 (30분) | 템플릿 복사 + 프로젝트별 값 채우기 (5분) |
| **신규 투입** | "지난 프로젝트 어떻게 했었지?" 동료에게 물어봄 | 이전 프로젝트 CLAUDE.md가 그대로 참고 자료 |

### 2.2 이 변화를 만든 세 가지 장치

**CLAUDE.md (프로젝트 컨텍스트)**
- 프로젝트 폴더에 CLAUDE.md를 두면, Claude Code가 매 세션마다 "이 프로젝트가 뭔지" 자동 인식
- 데이터 분석 규칙, 작업 플로우, 분석 구조까지 내장 → 같은 지시를 반복할 필요 없음
- 사람이 읽는 문서가 아니라 **시스템이 읽는 문서**. 팀원이 의식하지 않아도 Claude Code가 규칙을 따른다

**LESSONS.md (레슨런 축적)**
- 세션 종료 시 `/wrap` 스킬로 자동 기록. 현재 213줄 축적
- 데이터 분석 규칙, Notion 연동 팁, 설문 설계 레슨런, 교차분석 검증법 등
- **"같은 실수를 두 번 하지 않는" 시스템**

**프로젝트 템플릿 + 작업 가이드**
- 인터뷰 분석 7개 섹션, 설문 설계 체크리스트 등 반복 검증된 프로세스를 템플릿화
- Athena 테이블 구조, 흔한 쿼리 함정 등을 가이드 문서로 정리
- 프로젝트를 시작할 때 "어떻게 시작하지?"가 아니라 "여기서부터 시작하면 돼"

### 2.3 팀에 이미 있는 강점

팀 저장소에는 이미 강력한 자산이 있다.

| 영역 | 내용 | 상태 |
|------|------|------|
| **팀원** | evie, fred, haley, helena, ikpyo, ivy, rosie, sungbae (8명) | 활발 |
| **스킬 33개** | analyze-interview, analyze-survey, query, viz, notebook, hypothesis-analysis, dw-expert, fpna-analyst 등 | 다양하고 깊이 있음 |
| **분석 프로젝트** | 17개+ (렌탈고객분석, 홈지면CVR, 오프라인플래그쉽, 신선식품 리크루팅 등) | 활발 |

특히 데이터 분석 도구가 강하다:
- `query` 스킬 (7개 보조문서) — 쿼리 패턴, 메트릭 정의, 테이블 네이밍 컨벤션
- `viz` 스킬 (3개 보조문서) — 시각화 스타일/차트 가이드
- `hypothesis-analysis` 스킬 (6개 보조문서) — 코호트 쿼리, 리포트 템플릿

**부족한 것은 도구가 아니라 "연결"이다.** 좋은 스킬이 있지만, 그 스킬이 작동하는 맥락(CLAUDE.md)이 없고, 쓰면서 배운 교훈(LESSONS.md)이 기록되지 않고, 프로젝트를 시작하는 표준(템플릿)이 없다.

| 기능 | 나의 검증된 패턴 | 팀 기존 자산 | 관계 |
|------|------------|---------|------|
| 인터뷰 분석 | CLAUDE.md 기반 7섹션 프로세스 | `analyze-interview` | 보완 |
| 설문 분석 | 서베이몽키 MCP + 설계 레슨런 | `analyze-survey` | 보완 |
| 고객 분석 | `/customer-analysis` 스킬 | - | 신규 |
| 데이터 쿼리 | - | `query` (7개 보조문서) | 팀 유지 |
| 시각화 | - | `viz` (3개 보조문서) | 팀 유지 |
| 가설 분석 | - | `hypothesis-analysis` (6개 보조문서) | 팀 유지 |
| 글다듬기 | `/humanizer` 스킬 | - | 신규 |
| DW/FP&A/노트북 | - | dw-expert, fpna-analyst, notebook | 팀 유지 |

## 3. 어떻게 바꿀 것인가

핵심은 세 가지다: **시스템이 기억하게 만들고**, **프로젝트를 할수록 팀이 성장하는 구조를 만들고**, **노하우가 흐르는 파이프라인을 깔자.**

### 3.1 시스템이 기억하게 만들기 — 팀 CLAUDE.md

가장 임팩트 큰 단일 변경. 이 파일 하나로 Claude Code가 매 세션마다 팀의 맥락을 자동으로 인식한다.

**왜 필요한가**: 지금은 Claude Code가 매 세션마다 "이 저장소가 뭔지" 처음부터 추론한다. 팀이 어떤 규칙을 쓰는지, 어떤 실수를 조심해야 하는지 모른다. CLAUDE.md 하나면 이 문제가 사라진다.

**팀 CLAUDE.md 초안 (부록 A 참조)**에 포함할 내용:
- 저장소 개요 및 폴더 구조
- 데이터 분석 규칙 (event_type, GMV 기준 등)
- 코드 작업 플로우 (5단계)
- 응답 스타일 (간결함, 프레임워크 지양)
- 프로젝트 템플릿 경로 안내
- MCP 설정 가이드 참조

### 3.2 프로젝트를 할수록 팀이 성장하는 구조 — 템플릿 + 레슨런

**프로젝트 템플릿**: 검증된 프로세스를 복사해서 시작할 수 있는 표준

기존 팀 구조를 유지하면서 `_templates/` 폴더를 추가한다.

```
user-insight/
├── _templates/                    # [신규]
│   ├── interview/
│   │   └── CLAUDE.md.template    # 인터뷰 프로젝트 CLAUDE.md 표준
│   ├── survey/
│   │   └── CLAUDE.md.template    # 설문 프로젝트 CLAUDE.md 표준
│   └── data-analysis/
│       └── CLAUDE.md.template    # 데이터 분석 CLAUDE.md 표준
```

**왜 필요한가**: 지금은 프로젝트를 시작할 때 "어떻게 구조를 잡지?"부터 고민한다. 템플릿이 있으면 5분 만에 검증된 구조로 시작할 수 있고, 프로젝트가 끝난 CLAUDE.md는 다음 유사 프로젝트의 참고 자료가 된다.

인터뷰 CLAUDE.md 템플릿 핵심 (부록 B 참조):
- 프로젝트 개요 (핵심 질문, 대상, 방법)
- 분석 문서 7개 섹션 구조 (데이터 교차검증 포함)
- Athena 쿼리 (기본 구매, 할인율, 월별 추이)
- 진행 현황 테이블

**팀 레슨런 시스템**: 프로젝트를 할수록 팀이 똑똑해지는 구조

`.claude/LESSONS.md`를 팀 repo에 생성하고, `/wrap` 스킬로 세션 종료 시 자동 기록한다.

**왜 필요한가**: 지금은 실수에서 배운 교훈이 개인의 머릿속에만 남는다. LESSONS.md가 있으면 한 사람의 실수가 팀 전체의 학습이 된다. 프로젝트를 반복할수록 팀이 더 적은 실수를 하게 된다.

초기 시드 (나의 레슨런에서 팀 공유 적합분 추출):
- 데이터 분석 규칙 (event_type, GMV, 할인금액 부호)
- Notion 연동 팁 (회사/개인 구분, 토글 편집, 테이블 교체)
- 설문 설계 레슨런 13개 (척도 적합성, MECE, 이중질문 등)
- 교차분석 검증 (역검산, validate_crosstab)
- 가설 판정 Devil's Advocate 체크리스트

이후 팀원 모두의 세션에서 레슨런이 추가되며, 팀 공동의 지식 자산으로 성장한다.

### 3.3 노하우가 흐르는 파이프라인 — 가이드 + 스킬

**작업 가이드**: 반복적으로 부딪히는 함정과 해결법을 문서화

```
user-insight/
├── guides/                        # [신규]
│   ├── 커머스_고객분석_가이드.md   # Athena 테이블 구조, 쿼리 템플릿, 함정 목록
│   ├── 설문_분석_프로세스_가이드.md # 설문 분석 방법론
│   └── MCP_설정_가이드.md         # MCP 서버별 설정법 (Athena, SurveyMonkey, Notion 등)
```

**왜 필요한가**: `payment_type` 컬럼이 없다는 걸, `category_1depth_name`이 없다는 걸 매번 실행 오류로 배울 필요가 없다. 한 번 정리해두면 Claude Code가 가이드를 참조해서 처음부터 올바른 쿼리를 짠다.

**프로세스 공유 + 스킬 보강**

핵심은 **CLAUDE.md 템플릿에 프로세스를 내장**하는 것이다. 인터뷰 분석 7개 섹션, 데이터 교차검증 절차 등은 스킬이 아니라 프로젝트 CLAUDE.md에 적혀 있으면 Claude Code가 자연스럽게 따른다.

실제로 사용 중인 스킬 (추가 대상):

| 스킬 | 용도 | 비고 |
|------|------|------|
| `customer-analysis` | User ID → Athena 구매/행동 자동 분석 | 반복 사용 중 |
| `humanizer` | AI 글 티 제거 | 공유 전 마무리 단계에서 사용 |
| `wrap` | 세션 레슨런 자동 기록 | LESSONS.md 시스템의 핵심 |
| `gitgo` | 자동 커밋+푸시 | 팀 repo 전용으로 조정 |

기존 팀 스킬(`query`, `viz`, `hypothesis-analysis` 등)은 그대로 유지. 강점 영역이다.

| 고객 데이터 분석 1건 | 1시간 | 10분 | ~85% | `/customer-analysis`: Athena 쿼리 자동 생성+실행+해석 |
| VoC 리포트 1건 | 2시간 | 15분 | ~85% | CSV→노션 파이프라인 7회 운영 실측 |
| 프로젝트 시작 | 30분 | 5분 | ~85% | CLAUDE.md 템플릿 복사 + 프로젝트별 값 채우기 |

### 5.2 더 중요한 변화

**"누가 해도 같은 품질"**
- 7개 섹션 표준으로 분석 산출물 구조 통일
- 데이터 교차검증이 프로세스에 내장되어, 인터뷰 인사이트 ↔ 실제 데이터 괴리 자동 체크

**"같은 실수를 두 번 하지 않는 팀"**
- 한 사람이 발견한 함정이 LESSONS.md에 기록되고, 다음에 다른 사람이 같은 작업을 할 때 Claude Code가 자동으로 피함
- 프로젝트를 반복할수록 실수가 줄어드는 구조

**"프로젝트가 끝나도 지식이 남는 팀"**
- 완료 프로젝트의 CLAUDE.md가 아카이브로 남아, 유사 프로젝트 참조 가능
- 나의 레슨런이 시드가 되고, 팀원 모두의 레슨런이 추가되어 팀 공동 자산으로 성장

**"신규 팀원도 첫날부터 표준으로 작업"**
- CLAUDE.md + 템플릿이 있으면 신규 팀원도 첫 프로젝트부터 팀의 표준 프로세스로 작업 가능
- 개인 세팅 시간 대폭 단축

## 6. 리스크 및 대응

### 6.1 "기존 스킬/프로세스와 충돌하지 않나"

**대응**: 기존 스킬은 건드리지 않는다. 추가하는 스킬 4개(customer-analysis, humanizer, wrap, gitgo)는 팀에 없던 영역이라 충돌 없음. 인터뷰 분석 프로세스는 CLAUDE.md 템플릿으로 공유하므로 기존 `analyze-interview` 스킬과 별개로 작동.

### 6.2 "CLAUDE.md를 안 읽는 팀원"

**대응**: CLAUDE.md는 팀원이 읽는 문서가 아니라 Claude Code가 읽는 문서. 팀원은 의식하지 않아도 Claude Code가 자동으로 규칙을 따른다. 팀원이 해야 할 건 단지 "평소처럼 Claude Code를 쓰는 것"뿐.

### 6.3 "한 사람의 방식을 팀에 강요하는 것 아닌가"

**대응**: 이 제안은 표준 강제가 아니라 "검증된 기본값(default) 제공". 팀원은 자유롭게 수정 가능. `_templates/`는 참고용이지 강제가 아님. CLAUDE.md의 규칙도 팀 합의로 수정 가능. Phase 3에서 피드백을 받아 팀에 맞게 조정하는 것이 목적.

### 6.4 "유지보수 비용"

**대응**: LESSONS.md는 `/wrap` 스킬로 자동 기록. CLAUDE.md는 저장소 구조가 바뀔 때만 업데이트 (월 1회 미만 예상). 가이드 문서는 Athena 테이블 구조가 바뀔 때만 수정.
